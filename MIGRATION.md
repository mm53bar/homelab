# Migration from Portainer to Arcane

This document explains how this repository was created and the migration process from Portainer to Arcane.

## Background

This homelab was originally managed using **Portainer** on a Synology NAS. Portainer stored Docker Compose files in its internal database, making them difficult to version control and manage via Git.

The migration to **Arcane** was performed to enable:
- Git-based infrastructure as code
- Better version control of compose files
- Easier backup and restoration
- Automated sync from Git repository
- Simplified multi-environment management

## Migration Date

**January 2026**

## Original Structure

Portainer stored compose files in numbered directories:

```
/volume1/docker/compose/
├── 2/docker-compose.yml      (Plex)
├── 5/docker-compose.yml      (NZBGet)
├── 6/docker-compose.yml      (Radarr)
├── 7/docker-compose.yml      (Sonarr)
├── 12/docker-compose.yml     (Organizr)
...
├── 91/docker-compose.yml     (TRMNL)
```

This numbering system was auto-generated by Portainer and lacked semantic meaning.

## New Structure

The new Arcane-compatible structure organizes services by category:

```
projects/
├── media-management/     # *arr stack (Sonarr, Radarr, etc.)
├── media-servers/        # Plex, Jellyfin, Navidrome
├── downloads/            # NZBGet, qBittorrent, Transmission
├── books/                # Calibre, Calibre-Web, Kavita
├── dashboards/           # Organizr, Heimdall, Homepage
├── home-automation/      # Home Assistant, MQTT
├── personal/             # Paperless, Mealie, Wallabag, Forgejo
├── photos/               # iCloudPD, PhotoStructure, Immich
├── database/             # PostgreSQL, pgAdmin
└── utilities/            # Omada, YouTube-DL, etc.
```

Each service has its own directory with:
- `compose.yaml` (renamed from docker-compose.yml for Arcane compatibility)
- `.env` file (if needed)
- `README.md` (for documentation)

## Changes Made During Migration

### File Naming
- Renamed `docker-compose.yml` → `compose.yaml` (Arcane convention)
- Preserved `.env` files where they existed

### Docker Compose Syntax
- No changes to compose file syntax were required
- Arcane is fully compatible with Docker Compose v2 and v3 syntax

### Volume Paths
- All volume paths maintained as-is: `/volume1/docker/<service>/`
- No changes needed since Arcane runs on the same Synology NAS

### Networks
- Existing Docker networks (e.g., `media-network`) preserved
- Network definitions remain unchanged

### Environment Variables
- `.env` files preserved for Radarr and Radarr-SD
- All other environment variables kept in compose files

## Migration Steps Performed

1. **Backup**: Copied entire `/volume1/docker/compose/` directory from Synology
2. **Analysis**: Identified all 45+ services and their purposes
3. **Categorization**: Grouped services into logical categories
4. **Directory Structure**: Created new categorized directory structure
5. **File Migration**: Copied and renamed compose files to new locations
6. **Documentation**: Created README files for main categories
7. **Bulk Import Config**: Generated `arcane-sync.json` for one-click import
8. **Git Repository**: Initialized Git repo with proper structure

## Post-Migration Steps

After importing projects into Arcane:

1. **Verify Services**: Check that all containers started successfully
2. **Test Functionality**: Verify each service works as expected
3. **Configure Git Sync**: Set up automatic syncing from this repository
4. **Remove Portainer**: Once confident everything works, Portainer can be removed
5. **Update Documentation**: Keep README files current as services change

## Benefits Realized

- **Git-based workflow**: All changes tracked and versioned
- **Easy rollback**: Git history allows reverting changes
- **Documentation**: Category READMEs explain service purposes
- **Semantic naming**: Service names are meaningful, not numbered
- **Bulk operations**: Can update multiple services via Git push
- **Disaster recovery**: Git repo serves as full backup of infrastructure

## Rollback Plan

If needed to return to Portainer:

1. Portainer can import Docker Compose files from filesystem
2. Original numbered structure preserved in Git history
3. Volume data unchanged, so no data loss
4. Simply point Portainer at compose files

## Important Notes

### What Was NOT Changed

- Container data and volumes (all preserved)
- Port mappings (all kept the same)
- Network configurations
- Environment variables and secrets
- Volume mount paths

### What WAS Changed

- File organization (numbered → categorized)
- File naming (docker-compose.yml → compose.yaml)
- Management tool (Portainer → Arcane)
- Version control (none → Git)

### Services with Special Configurations

- **Radarr & Radarr-SD**: Have `.env` files
- **Home Assistant**: Single compose file contains both Home Assistant and MQTT
- **iCloudPD**: Single compose file contains two instances (mike and sheila)
- **PostgreSQL**: Single compose file contains PostgreSQL and pgAdmin
- **Paperless-NGX**: Multi-container setup with Redis

## Future Maintenance

Going forward, all infrastructure changes should:

1. Be made in this Git repository
2. Be committed with descriptive messages
3. Be pushed to trigger Arcane auto-sync
4. Be documented in README files when appropriate

## Questions for Future You

**Q: Why is service X in category Y?**
A: Services were categorized by primary function. Some services span categories.

**Q: Can I rename services?**
A: Yes, but you'll need to update `arcane-sync.json` and re-import in Arcane.

**Q: What if I want to add a new service?**
A: Create a directory in the appropriate category, add compose.yaml, commit, and import in Arcane.

**Q: How do I update a service?**
A: Edit the compose.yaml in Git, commit, push. Arcane will auto-sync and redeploy.

**Q: Where are the original Portainer files?**
A: Preserved in the `compose/` directory for reference (not imported to Arcane).

## Conclusion

This migration modernizes the homelab infrastructure while preserving all functionality. The categorized structure, documentation, and Git-based workflow will make future maintenance much easier.
